# Couple Retrospective Adversarial Workflow

## When to Use
- 有情侣冲突案例需要深度对抗分析时
- 用户要求"两个人直接对话，不要笔杆子+参谋"时
- 需要超过6分钟的完整对抗过程时

## 前置条件
1. couple-retrospective skill 已加载（了解分析框架：五层心智、打分、行动承诺）
2. 有冲突故事素材（真实情侣 / 文学IP / 用户描述均可）

## 执行流程

### Step 0：确认素材
用户提供故事片段或角色设定。哪怕是"林黛玉 vs 贾宝玉"这种文学IP也能跑通流程。

### Step 1：建立对抗结构

每轮对抗由**两个代理同时生成各一段台词**组成：
- Agent A：生成自己的台词
- Agent B：生成自己的台词
- 两段合在一起为一轮完整对话

不要让一个代理"生成两个人的台词"——那样失去了对抗性。

### Step 2：分批驱动子代理（关键！）

**delegate_task的goal格式优先**（batch格式有bug）：
```
tasks=[
  {"goal": "你是宝玉。第N轮台词...", "context": "背景", "toolsets": [...]},
  {"goal": "你是黛玉。第N轮台词...", "context": "背景", "toolsets": [...]}
]
```
- 每批2个子任务（同时运行）
- 单次超时设置充足（宝玉约5秒，黛玉约30秒）
- 每个子代理是独立会话，不知道另一方的台词——需要主代理"居中传话"来构建对抗感

### Step 3：追踪轮次

每轮需要把上一轮两个代理的台词都放进下一轮的context里，这样代理能感知对话发展而不是重复论点。

格式：
```
第N-1轮：
🎙️ 宝玉："..."
🎙️ 黛玉："..."

第N轮（最终陈述）...
```

### Step 4：达到充分对抗

**最少8轮**（红楼梦测试12轮约5-6分钟）。建议：
- 前6轮：基础冲突展开
- 第7-9轮：深入/反转
- 第10轮+：收束/最终陈述

### Step 5：输出分析报告

报告结构（五层心智框架）：

```
## 五层心智定位

Layer 0（身份）：
  宝玉：...
  黛玉：...

Layer 1（表达DNA）：
  宝玉：关键台词模式
  黛玉：关键台词模式

Layer 2（心智模型）：
  宝玉：爱 = ...
  黛玉：爱 = ...

Layer 3（决策启发式）：
  宝玉：被质疑 → ...
  黛玉：被冷落 → ...

Layer 4（反模式）：
  宝玉：...
  黛玉：...

Layer 5（诚实边界）：
  宝玉触碰了：...
  黛玉触碰了：...
```

打分矩阵：
```
宝玉总分：X/10
  情绪控制：X/10
  沟通质量：X/10
  踩雷清单：X/10
  修复态度：X/10
  关系增益：X/10

黛玉总分：X/10
  （同上五项）

双方均 ≥ 6分 → 通过
```

行动承诺：每方2条具体承诺。

## 踩坑记录

1. **xiaohongshu所有入口被封**
   - WSL IP被封（"请求异常"/验证码）
   - browser访问被封
   - xhs cli需要cookie（用户本地Chrome已登录才有用）
   - 解决：改用文学IP（如红楼梦）测试流程，或让用户提供具体故事

2. **delegate_task batch格式bug**
   - batch格式有时只跑一个任务
   - 解决：改用goal格式单任务

3. **黛玉代理超时问题**
   -黛玉角色生成台词较慢（约30秒），宝玉约5秒
   - 解决：分开设置timeout，宝玉5-10秒，黛玉30-60秒

4. **12轮刚好踩线**
   - 12轮约5-6分钟
   - 要超过6分钟建议跑14-16轮
