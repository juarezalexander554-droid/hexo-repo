# 微信聊天记录读取器

## 作用

从已解密的微信数据库中提取**特定联系人**的聊天记录，输出：
1. 标准化时间线（格式化对话）
2. 行为分析数据（发言比例、回复延迟、情绪转折、沉默窗口）
3. 依恋行为标记（焦虑型/回避型信号自动识别）
4. 结构化 JSON + Markdown 报告

输出直接供 `couple-retrospective` 主框架作为 **Step 0 素材** 使用。

## 快速开始（Step by Step）

### Step 1：提取 Key + 解密数据库

```python
import sys, os, json

# 切换到工具目录
TOOLS_DIR = os.path.dirname(os.path.abspath(__file__))
sys.path.insert(0, TOOLS_DIR)
os.chdir(TOOLS_DIR)

from wxManager.decrypt.get_wx_info import read_info
with open('./wxManager/decrypt/version_list.json', encoding='utf-8') as f:
    vl = json.load(f)

result = read_info(vl)
key = result[0]['key']
wxid = result[0]['wxid']
print(f'Key: {key}, wxid: {wxid}')

# 解密数据库
from wxManager.decrypt.decrypt_v3 import decrypt_db_files
src = fr'D:\\Documents\\WeChat Files\\{wxid}\\Msg'
out = r'D:\\Claude_Notes\\wechat_decrypted'
os.makedirs(out, exist_ok=True)
decrypt_db_files(key, src, out)
print('解密完成！')
```

**成功标志：** `errcode=200`，输出 `MicroMsg.db` 和 `Multi/MSG0.db`

### Step 3：提取聊天记录


## 数据库结构

### MicroMsg.db · Contact 表

| 字段 | 说明 |
|------|------|
| `UserName` | wxid（如 `wxid_xxxxx`） |
| `Alias` | 微信号（自定义英文ID） |
| `Remark` | 备注名 |
| `NickName` | 微信昵称 |

### MSG0.db · MSG 表

| 字段 | 说明 |
|------|------|
| `CreateTime` | Unix 时间戳（秒级） |
| `Type` | 消息类型（1=文本，3=图片，34=语音，43=视频，10000=系统消息） |
| `IsSender` | 1=我发送的，0=对方发送的 |
| `StrContent` | 消息文本内容 |
| `TalkerId` | 关联 Name2ID 表的 _id |

### 消息类型编码

| Type | 类型 |
|------|------|
| 1 | 文本 |
| 3 | 图片 |
| 34 | 语音 |
| 43 | 视频 |
| 47 | 表情 |
| 49 | 链接/XML/小程序 |
| 10000 | 系统消息（撤回提示等） |

## 执行流程

### Step 0：确认数据源

用户提供：
- 解密数据库路径（默认 `D:\Claude_Notes\wechat_decrypted_v2\`）
- 联系人昵称或备注名

### Step 1：查找 wxid

```sql
SELECT UserName, Remark, NickName 
FROM Contact 
WHERE NickName LIKE '%目标昵称%' OR Remark LIKE '%目标昵称%';
```

### Step 2：提取聊天记录

```sql
SELECT
    m.CreateTime,
    m.IsSender,
    m.Type,
    m.StrContent
FROM MSG m
LEFT JOIN Name2ID n ON n._id = m.TalkerId
WHERE n.UsrName = 'wxid_xxx'
  AND m.Type = 1
ORDER BY m.CreateTime ASC;
```

### Step 3：格式化时间线

```
[YYYY-MM-DD HH:MM] [发送方] 内容
```

### Step 4：行为分析（自动）

#### 4.1 时间维度

| 指标 | 计算方法 | 复盘价值 |
|------|---------|---------|
| 消息总量 | 总条数 | 亲密度基准线 |
| 日均消息量 | 总数 / 天数 | 日常连接频率 |
| 峰值日期 | 单日最多消息数 | 重要事件锚点 |
| 深夜占比 | 00:00-05:00 消息 / 总消息 | 生活节奏匹配度 |
| 沉默窗口 | 连续 >4h 无消息 | 回避行为信号 |

沉默窗口严重度：>24h = 严重，>48h = 危险

#### 4.2 情绪维度

**情绪关键词分层统计：**

| 层级 | 关键词示例 | 统计方式 |
|------|-----------|---------|
| 正向 | 喜欢、爱你、想你、哈哈 | 按发送方计数 |
| 负向 | 算了、不想、别烦、滚 | 按发送方计数，附上下文 |
| 追逐信号 | 你在干嘛、怎么不回、睡了吗 | 焦虑型行为标记 |
| 回避信号 | 在忙、晚点、先不聊、困了 | 回避型行为标记 |

**情绪转折点识别：**
- 连续3条消息情绪标签翻转（正→负 或 负→正）
- 沉默窗口后第一条消息带强烈情绪词
- 某关键词在10分钟内出现2次以上

#### 4.3 沟通模式维度

| 指标 | 说明 |
|------|------|
| 发言比例 | 双方各发了多少条，差距多大 |
| 回复延迟分布 | 平均多久回、中位数、即时回复率 |
| 典型开场/收尾句式 | 每人常用的第一句话和最后一句话 |
| 单字敷衍频率 | "嗯""哦""行""好"的计数 |

**关键洞察示例：**
- 团团平均回复 42 分钟，壮壮 8 分钟 → 感知差距 5 倍
- 团团深夜消息多 → 情绪在夜间暴露
- 壮壮单字敷衍 45 次 → 回避型信号

#### 4.4 轮次分析（核心）

将连续对话按**轮次**切分（A 发言 → B 回应 = 1 轮），逐轮评估：

| 轮次评估维度 | 说明 |
|-------------|------|
| 情绪走向 | 本轮情绪是升温 / 持平 / 降温 |
| 回应质量 | 是否接住了对方的情绪/事实 |
| 雷点触发 | 本轮是否踩雷（说了不该说的话） |
| 修复行为 | 本轮是否有道歉、解释、示弱等修复动作 |
| 话题层级 | 事实层 → 情绪层 → 需求层，是否对齐 |

**轮次级输出示例：**
```
第 7 轮 | 2026-04-06 03:12
团团："不能闲，停下来闭眼就是你"
壮壮："要脸干嘛要你就行了"
情绪走向：↑ 升温（试探 → 直球接纳）
回应质量：高（团团暴露脆弱，壮壮直接接住）
雷点：无
话题层级：情绪层 → 情绪层（对齐）
```

**关键洞察：** 复盘不是看"发了多少条"，而是看"第几轮崩掉的、为什么崩、哪一轮本可以救回来"。

#### 4.5 依恋行为标记

**焦虑型信号（自动检测）：**
- 同一问题被重复问 2 次以上
- "你会走吗""你还在吗""真的吗"等确认句式
- 主动求助频率（"帮我""怎么办"）

**回避型信号（自动检测）：**
- 超时回复（>2h）次数
- 话题回避（从感情话题切到工作/游戏/睡觉）
- 单字回复占比

**追逃循环检测：**
- 追逐方试探 → 回避方沉默/敷衍 → 追逐方爆发/放弃
- 统计循环次数、典型触发话题

### Step 5：论文级分析（核心）

基于 Paper A/B/C 理论框架，对聊天记录进行深度解读，输出带论文框架的分析结论。

#### Paper A：追逐-逃避循环

识别对话中是否出现追逐-逃避循环：

| 检查项 | 判定标准 |
|--------|---------|
| 循环启动轮次 | 哪一轮出现了"追逐方发出信号 → 回避方沉默/敷衍" |
| 循环加速节点 | 哪一轮追逐方开始升级信号（质问/情绪爆发） |
| 循环断裂节点 | 哪一轮对话彻底崩掉或意外修复 |
| 双方角色 | 谁是追逐方、谁是回避方（可能互换） |

**输出示例：**
```
追逐-逃避循环检测：✅ 发现 2 次循环

循环 #1 | 第 12-18 轮
- 启动：第 12 轮，团团问"你在干嘛"（试探信号）
- 加速：第 15 轮，团团连发 5 条消息（信号升级）
- 断裂：第 18 轮，壮壮说"算了"（逃避方关闭对话）
- 角色：团团=追逐方，壮壮=回避方

循环 #2 | 第 31-35 轮
- 启动：第 31 轮，壮壮问"你是不是不在乎了"
- ...（团团变为回避方，壮壮变为追逐方，角色互换）
```

#### Paper B：结构性冲突

判断冲突性质是**沟通问题**还是**系统设计冲突**：

| 维度 | 沟通问题 | 结构性冲突 |
|------|---------|-----------|
| 双方需求 | 可以被同时满足 | 本质上互相排斥 |
| 解决方式 | 换个说法就行 | 需要系统级妥协 |
| 典型表现 | "你语气不好" | "我要空间，你要亲密" |

**输出示例：**
```
结构性冲突诊断：✅ 结构性冲突

团团的核心需求：自主/空间/不被追问
壮壮的核心需求：被回应/被记住/被确认

结论：这不是"谁不会说话"，是两个操作系统在报错。
改进方向不是"团团多说点"，而是建立"最低确认信号"机制。
```

#### Paper C：EWDM 期望-撤退动态模型

追踪对话中期望（E）和撤退（W）的变化曲线：

```
E（期望）↑ → 追逐行为 → W（撤退）↑ → 循环加速
                ↓
        打破循环的节点 = 某一方停止追逐/撤退
```

**输出示例：**
```
EWDM 曲线分析：

第 1-5 轮：E 平稳，双方正常交流
第 6 轮：壮壮 30 分钟未回 → E(团团) ↑（期望上升）
第 8 轮：团团追问"人呢" → 追逐行为启动
第 10 轮：壮壮回复"在忙" → W(团团) ↑（失望/撤退感）
第 12 轮：团团情绪爆发 → E(壮壮) ↑（对团团的反应产生压力期望）

打破循环的潜在节点：
- 第 10 轮，如果壮壮说"在忙，晚点聊"而非"在忙"，E(团团) 不会继续上升
- 第 12 轮，如果团团停止追问，给壮壮空间，循环在此断裂
```

## 踩坑记录

1. **Key 有效期**：Key 从微信进程内存提取，**每次微信重启后必须重新提取**。
2. **时间戳单位**：Windows 微信 CreateTime 是秒级 Unix 时间戳。若数值 > 1e12 则为毫秒，需除以 1000。
3. **Emoji 干扰**：关键词统计前需去除 Emoji，否则匹配失败。
4. **长消息拆分**：相同 CreateTime 的连续消息可能是一条被拆分的长消息，需合并。
5. **只读查询**：查询数据库时使用 `mode=ro` URI，避免修改原始数据。
